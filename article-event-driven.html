Why I Switched to Event-Driven Architecture for Scalable Microservices

September 22, 2025

In the early stages of building AlgoMentor, I used direct synchronous HTTP calls between services. As traffic grew to 500+ concurrent users, this tight coupling led to cascading failures and high latency.

The Shift to Asynchronous Messaging

I refactored the system to use an event-driven approach with Redis Pub/Sub (and later Kafka for durability). Instead of Service A waiting for Service B to finish, Service A publishes an event and moves on.

Benefits Observed

  • Decoupling: Services can be deployed and scaled independently.
  • Resilience: If the "Code Executor" service is down, the "Submission" service can still accept user code and queue it for later.
  • Responsiveness: The user gets immediate feedback ("Submission Received") while the heavy processing happens in the background.

Conclusion

While event-driven architecture adds complexity in tracing and debugging, the gains in scalability and fault tolerance are non-negotiable for high-volume systems.
